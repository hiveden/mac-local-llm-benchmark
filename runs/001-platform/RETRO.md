## 数据跑了 4 轮的原因

| 轮次 | Git Commit | 发现的问题 | 修复方式 |
|------|-----------|-----------|---------|
| 第一轮 | 280b59d | Ollama KV cache + mlx-lm prompt cache → TTFT 虚低 | 加 inter_round_cleanup |
| 第二轮 | bd7256a | mlx-lm prompt cache 仍在（第一轮未发现） | 加 --prompt-cache-size 0 + 重启 server |
| 第三轮 | aab802f | mlx-lm 重启 server → 1.4s JIT 开销 → TTFT 虚高 | 改回不重启（cache 已在启动时禁用） |
| 第四轮 | 8486b87 | ✅ 数据干净 | 只重跑 mlx-lm，Ollama/oMLX 不动 |

**根因**: 设计阶段没有调研三个平台的缓存机制。如果一开始手动 curl 同一 prompt 两次观察 TTFT 变化，30 秒就能发现所有缓存行为。

## 核心失误

### 1. 技术选型：Bash + 内嵌 Python

选了 Bash 做主循环 + 内嵌 Python 做 JSON/SSE 解析。导致变量注入、引号地狱、调试困难等问题反复被审核出来。10+ 轮代码审核中大部分问题来自这个选型。

**教训**: 测试/自动化脚本优先用纯 Python。

### 2. 没有小步验证

先写完 6 个场景的完整架构、脚本、文档，再一次性跑全量测试。发现问题时改动面已经很大。

**应该做的**:
1. 手动 curl 三个平台各一次 → 确认 API 能通
2. 同一 prompt 连发两次 → 发现缓存行为
3. 写一个 provider 跑 3 轮 → 验证数据格式
4. 再加第二个 provider → 验证跨平台对比
5. 最后加齐所有场景 → 完整数据

### 3. 修改已有脚本而非新增场景

前三轮直接改 lib.sh 重跑，覆盖了历史数据。违反了"场景独立、已有数据不动"的原则。

**教训**: 发现问题后应该新增独立场景补充数据，不改已有脚本。第四轮做对了（只重跑 mlx-lm）。

### 4. 并行跑共享资源的测试

两批测试并行跑，共享 Ollama/oMLX 服务，cleanup_environment 互相卸载对方的模型，数据全部污染。

**教训**: 测试必须串行执行（已写入 DECISIONS.md #13）。

### 5. 测试期间改代码

用户在跑测试时我修改了脚本文件，导致数据不一致。

**教训**: 测试过程中绝对不能修改任何脚本（已写入 CLAUDE.md）。

## 做对的事

1. **场景独立架构** — 让第四轮只重跑 mlx-lm 成为可能
2. **数据进 git** — 四轮数据都可追溯（commit hash 记录在 KNOWN_ISSUES.md）
3. **多轮审计** — 每轮发现真实问题并修复，没有带着错误数据出结论
4. **prefill 差值验证** — 三平台 A1→A3 差值 ~1230ms 一致，证明第四轮数据干净
5. **KNOWN_ISSUES 文档化** — 8 个 DA 问题完整记录，别人看到数据就知道怎么解读
6. **lib.sh 共享函数** — 减少 62% 代码重复，修 bug 改一处

## 最终数据结论

| 指标 | oMLX | Ollama | mlx-lm |
|------|------|--------|--------|
| A1 TTFT (短 prompt) | **87ms** | 133ms | 204ms |
| A3 TTFT (长 prompt) | **1317ms** | 1455ms | 1619ms |
| A1 总耗时 | 1795ms | 1662ms | **1293ms** |
| A2 总耗时 | 16109ms | 15699ms | **11710ms** |
| E2 Gemma4 TTFT | **408ms** | 2655ms | — |
| E2 Gemma4 总耗时 | **3740ms** | 7659ms | — |

- **oMLX prefill 最快**（所有场景 TTFT 最低）
- **mlx-lm decode 最快**（总耗时最短）
- **Gemma4 跨引擎**: MLX (oMLX) prefill 比 llama.cpp (Ollama) 快 6.5x
- **tok/s**: 仅 Ollama 可信 (~66 tok/s)，oMLX/mlx-lm 流式不返回 token 数
