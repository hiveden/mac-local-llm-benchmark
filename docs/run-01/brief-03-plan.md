四、测试方案（harness 化性能测试）
测试对象
主测：Qwen3.5-35B-A3B（三方都走 MLX 路径，最公平）
副测：Gemma4-26B（Ollama 走 Metal，展示 MLX 加速缺失的真实差距）
运行时对比



| 工具         | 模型来源                               | 路径                |
| ------------ | -------------------------------------- | ------------------- |
| Ollama 0.19  | `qwen3.5:35b-a3b-nvfp4`（已下载 21GB） | Ollama 原生 MLX     |
| mlx-lm serve | HF `mlx-community/Qwen3.5-35B-A3B-*`   | Apple 官方 MLX 基线 |
| oMLX         | 复用同一份 HF 模型                     | 社区 MLX 封装       |

测试任务（3 个场景）



| 任务类型   | Prompt 示例                             | 测什么           |
| ---------- | --------------------------------------- | ---------------- |
| 短问答     | “用一段话解释 RAG 的工作原理”           | TTFT、短文本速度 |
| 代码生成   | “用 Python 写一个 RAG 向量检索最小实现” | 中等长度生成     |
| 长文本总结 | 2000 token 技术文档 → 200 字总结        | 长 prompt 处理   |

每个任务 × 3 个工具 × 10 轮 = 90 次请求
核心指标
	∙	TTFT（首 token 延迟）：中位数，去掉冷启动第 1 轮
	∙	tok/s（解码速度）：中位数
	∙	tok/s 稳定性：标准差 / 变异系数
	∙	内存占用：峰值 / 平均
	∙	输出质量：人工抽检 3-5 个样本

五、重要声明（必须在视频和文档里说明）
量化差异陷阱
三个工具跑的不是同一份文件：
	∙	Ollama: NVFP4 量化
	∙	MLX 社区: group-wise 4bit 量化
	∙	速度差异 = 工具封装层差异 + 量化算法差异（不可拆分）
在视频里明说：这本身就是”AI 调研陷阱”主题的实例。
测试边界（DISCLAIMER）
这份数据能回答：
	∙	三个工具在 M4 Pro 64GB + 这些模型 + 这些任务的性能差异
这份数据不能回答：
	∙	其他 Mac 配置的表现
	∙	其他模型（Llama/DeepSeek 等）
	∙	长时间（4 小时+）运行
	∙	量化精度对输出质量的影响
	∙	中文 vs 英文任务差异
